\subsection{Congestion Control general Information}

Congestion Control describes all algorithms, that are used to control how much packet a sender is sending. 
The main focus of Congestion Control lies on TCP connections, because there a low latency and high throughtput a wanted, while the packet loss should be zero.
In \cite{JainsCongestionControl} describes the main reasons and challanges for congestion control algorithms that have not changed since.
The author identifies three main hardware limits of a bottleneck router, which are buffer, transmission speed and speed of CPU of the bottleneck router.
If a router is upgraded, but not proportionaly equal in all three at the same time, it will cause problems and actually limit the capacity of the adjusted router.
Therefore, Congestion Control algorithms are necessary, because Hardware has its limits.

\subsubsection{Measuring Indices}

When a Congestion Control Algorithm is developed, its founders want to know, if there idea is good or not. 
To determine a good Congestion Control algorithm there are a few indices.
One of them is Jains fairness Index and it measures the fairness of an congestion control algorithm in respect to how fair flows share the bottleneck resources.
Jains Index is a number between 0 and 1 where its value describes the amount in percentage of users in who get a fair share of the resources. 
So 1 means 100\% of the users get a fair share\cite{JainsIndex}.

\subsection{TCP Cubic}

When the internet startet to grow and it became evident, that it will be more high speed and long distance network, older TCP Congestion Control Algorithm have less scalibility to tackle the challenges of the internet\cite{TCPCUBIC}.
The authors of TCP Cubic wanted to build a stable and scalable TCP version that can handle the new challanges of the internet.
The basic idea behind TCP Cubic is that it uses a Cubic congestion window function\cite{TCPCUBIC}.
Most other Congestion Control Algorithms at that time increased their packet rate only by one packet per RTT, which could lead to a very slow start and before a sender would start fully using its network resources, it could take hours.
The cubic function starts to increase its packet rate very fast and then slows down in after some time, in order to not heavily overstep the bottleneck threshold.
\cite{TCPCUBIC}
It detects the bottlenecks limit by watching for packets loss. 
This means, when the packet loss increases, then the bottleneck is not able to handle more packets and TCP Cubic knows it has reached the limit.
\cite{TCPCUBIC}
